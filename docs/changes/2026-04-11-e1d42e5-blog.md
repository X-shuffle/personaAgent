# 博客版详解：记忆检索加上相似度阈值过滤（e1d42e5）

## 背景
在会话记忆检索里，仅靠 `topK` 通常会把“最像的几个”都拿回来，但“最像”不等于“足够相关”。
当候选整体相关度偏低时，仍会把噪声记忆拼进提示词，导致回复出现轻微跑偏。

这次改动的目标很明确：
- 在不破坏现有默认行为的前提下，增加一个可配置的“最低相似度门槛”；
- 低于门槛的记忆直接丢弃，提升上下文纯度。

## 改动点拆解

### 1) 新增环境配置项
文件：`.env.example`
- 新增：`MEMORY_SIMILARITY_THRESHOLD=0`

这保证了配置语义可见，默认值为 `0` 表示“不启用阈值过滤”，即与历史行为一致。

### 2) 配置结构扩展 + 边界收敛
文件：`internal/config/config.go`
- `Config` 增加字段：`MemorySimilarityThreshold float64`
- `envConfig` 增加字段：`MEMORY_SIMILARITY_THRESHOLD`（默认 `0`）
- `Load()` 中将阈值夹紧到 `[0,1]`
  - `< 0` => `0`
  - `> 1` => `1`

这样做的意义：避免非法配置穿透到运行时逻辑，降低异常配置导致的不可预期行为。

### 3) 启动阶段参数透传
文件：`cmd/server/main.go`
- `memory.NewService(...)` 在 `qdrant` / `inmem` 两个分支都新增传参：`cfg.MemorySimilarityThreshold`

此处只做透传，不改 memory mode 分支逻辑，回归风险低。

### 4) 记忆服务增加过滤逻辑
文件：`internal/memory/service.go`
- `DefaultService` 新增字段：`minSimilarity`
- `NewService(...)` 新增参数：`minSimilarity`
- 构造函数内再次对 `minSimilarity` 做 `[0,1]` 夹紧（双保险）
- `Retrieve(...)` 遍历命中结果时新增：
  - `if m.Score < s.minSimilarity { continue }`

最终效果：
- 仍按既有 `topK` 检索候选；
- 但只把“分数达标”的候选写入返回记忆。

### 5) 单测补齐
文件：`internal/config/config_test.go`（新增）
- 测试阈值读取：`0.77`
- 测试阈值夹紧：`-1 -> 0`、`2 -> 1`

文件：`internal/memory/service_test.go`
- 适配 `NewService` 新签名
- 新增阈值过滤测试：低分样本被过滤，仅保留高分样本

## 关键代码路径
- 配置入口：`internal/config/config.go`
- 服务装配：`cmd/server/main.go`
- 过滤执行：`internal/memory/service.go`
- 配置测试：`internal/config/config_test.go`
- 检索行为测试：`internal/memory/service_test.go`

## 行为变化

### 变更前
- 召回结果主要受 `topK` 控制。
- 即便分数较低，只要在前 `K` 名，就可能进入提示词上下文。

### 变更后
- 召回结果受 `topK + similarity_threshold` 双约束。
- 低于阈值的记忆被过滤，不参与后续提示构建。

### 默认兼容性
- 默认阈值 `0`，行为与历史版本一致（等价于不过滤）。

## 测试验证
- 配置层：验证环境变量解析与边界夹紧。
- 业务层：验证检索结果按阈值过滤。
- 改动集中在配置解析、服务构造和检索后处理，未引入额外 I/O 或外部依赖。

## 潜在问题与后续优化
1. **阈值过高导致“无记忆”**
   - 现象：上下文记忆命中率下降，回答可能更“短期记忆化”。
   - 建议：从低值开始（如 `0.1~0.3`）逐步调参。

2. **全局阈值粒度较粗**
   - 当前是全局配置，无法按会话、场景、用户动态调整。
   - 后续可考虑：
     - 按会话状态动态阈值；
     - 按 memory type 设置差异化阈值；
     - 增加召回分数分布日志用于运营调优。

3. **双处夹紧属于防御性重复**
   - 配置层与服务层都做了 `[0,1]` 限制，保证即使非配置路径构造服务也安全。
   - 若后续统一构造入口，可评估是否收敛到单点校验。
