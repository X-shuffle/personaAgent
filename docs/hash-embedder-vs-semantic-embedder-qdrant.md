# HashEmbedder、语义 Embedding 与 Qdrant：从原理到落地

这篇文章专门对齐你当前项目场景：

- 为什么会出现“AI 选品”召回到“打球”
- HashEmbedder、语义 Embedding、Qdrant 各自负责什么
- 为什么先换 Embedder，而不是先“改 Qdrant”
- BM25/TF-IDF 和语义检索到底怎么配合

---

## 1. 先看全链路：三层职责不要混

```text
[文本]
  │
  ▼
[Embedder: 把文本变向量]
  │
  ▼
[Vector DB(Qdrant/Chroma): 存向量 + 相似检索]
  │
  ▼
[LLM: 读召回上下文并生成回答]
```

- **Embedder** 决定“向量是否表达语义”
- **Qdrant** 决定“检索效率、过滤能力、索引能力”
- **LLM** 决定“最终回答表达与推理”

一句话：**向量质量看 Embedder，检索速度看 Qdrant，回答效果看 LLM。**

---

## 2. 你当前项目为什么会“串话题”

你当前链路是：

```text
对话文本 -> HashEmbedder -> Qdrant -> TopK + threshold -> 注入 Memories -> LLM 回答
```

问题根因通常是两层叠加：

1. 同一 `session_id` 下历史记忆都会进入候选池
2. HashEmbedder 不是语义模型，近邻关系可能跑偏

所以会出现：问“AI 选品”，结果把“打球”也召回并注入 `Memories:`，LLM 就顺着偏题内容回答。

## 3. HashEmbedder 是什么（以及为什么它“怪怪的”）

### 3.1 本质

- 思路：文本先做哈希，再映射成固定维度向量
- 优点：快、便宜、无外部依赖
- 缺点：**不是语义模型**，更像字符串指纹

可视化：

```text
文本A: "做 AI 选品软件能赚钱吗"
文本B: "AI 选品 SaaS 有商业价值吗"

语义上：很近
HashEmbedder 空间：不一定近（字面差异会影响很大）
```

### 3.2 你项目内的实现机制

```text
文本
  │
  ▼
SHA256 32字节: [b0 b1 ... b31]
  │
  ▼
映射到 dim 向量：v[i % dim] += b
  │
  ▼
L2归一化：v = v / ||v||
```

在你当前配置（`MEMORY_VECTOR_DIM=256`）下：

```text
i 只会是 0..31
i % 256 = i
=> 只有前 32 维被写入，后 224 维基本为 0
```

形态示意：

```text
[ x x x x x x x x x x x x x x x x x x x x x x x x x x x x x x x x | 0 0 0 ... 0 ]
```

### 3.3 用你的句子走一遍

输入：

```text
你觉得做一个 ai 选品软件能赚钱吗
```

流程：

1. 先 `sha256` 得到 32 字节 `b0..b31`
2. `i` 取值 `0..31`（不是小数 0.31）
3. 写入 `v[i % 256]`，等价于写 `v[0..31]`

结果：

```text
v = [x0, x1, ... x31, 0, 0, ... 0]  // 总长256
```

### 3.4 为什么是“32 字节”和“256 维”

```text
32 字节 <- SHA-256 算法定义（256 bit）
256 维  <- 项目配置 MEMORY_VECTOR_DIM
```

不是一回事：一个是算法输出长度，一个是工程向量维度。

### 3.5 把 MEMORY_VECTOR_DIM 改成 32 会怎样？

会“填满”32维向量，但仍是哈希指纹，不会自动变成语义向量。

注意三点：

1. 这是 **32维**，不是 32bit
2. 填满 ≠ 语义更强
3. 改维度会影响 Qdrant collection，可能要重建

## 4. 常见 Embedding 路线对比

### A) 云端语义 Embedder（API）

例如：OpenAI `text-embedding-3-small`

- 优点：效果稳定、接入快
- 缺点：按量付费、依赖外网

### B) 本地语义 Embedder（开源）

例如：`bge-large-zh`、`bge-m3`、`e5`

- 优点：本地可控、无按次费用、中文效果好
- 缺点：需要部署与算力

### C) 稀疏检索（BM25/TF-IDF）

严格说不是 dense embedding，但常用于 hybrid。

- 优点：关键词命中强、可解释
- 缺点：语义泛化弱（同义改写容易漏）

---

## 5. BM25 / TF-IDF 你可以这样理解

### 5.1 一句话直觉

```text
TF-IDF / BM25 不做语义编码，而是做关键词打分。
```

关注三件事：

- 在当前文档出现几次（TF）
- 在全库是否太常见（IDF）
- 文档是否过长（长度归一）

### 5.2 TF-IDF 入门图

```text
Query: "AI 选品 软件 赚钱"

D1: "AI 选品 软件 赚钱 路径"
D2: "今天 打球 很 开心"

D1 命中词更多，且命中词更关键 -> 分更高
D2 几乎不命中 -> 分更低
```

简化式：

```text
score(D, Q) ≈ Σ [ TF(term, D) × IDF(term) ]
```

### 5.3 BM25 为什么更常用

BM25 可以看作 TF-IDF 的工程增强：

- 词频收益“边际递减”
- 加入文档长度归一
- 在真实检索里通常更稳

示意：

```text
关键词出现次数: 1 -> 2 -> 3 -> 20
BM25增益:       快速上升后趋缓（不会无限线性加分）
```

### 5.4 稀疏 / Dense / 语义，最容易混的点

```text
┌───────────────┬────────────────────────────┬──────────────────────────────┐
│ 类型          │ 主要依据                    │ 擅长场景                      │
├───────────────┼────────────────────────────┼──────────────────────────────┤
│ 稀疏(BM25)    │ 关键词是否出现、出现多少      │ 型号/术语/数字等精确命中       │
│ Dense(形态)   │ 文本被编码为实值向量          │ 向量检索框架（不保证语义）      │
│ 语义Embedding │ 在 dense 向量中表达“意思接近” │ 同义改写、自然语言变化          │
└───────────────┴────────────────────────────┴──────────────────────────────┘
```

关键句：

```text
dense 是表示形式，semantic 是表示质量。
```

## 6. 为什么很多系统做 Hybrid（BM25 + 语义向量）

因为两边互补：

- BM25 负责关键词精准召回
- 语义向量负责改写鲁棒召回

常见流程：

```text
                    ┌──────────────┐
Query ─────────────►│  Query分析层  │
                    └──────┬───────┘
                           │
            ┌──────────────┴──────────────┐
            ▼                             ▼
   稀疏召回（BM25）                 向量召回（Semantic）
   TopN_sparse                      TopN_dense
            └──────────────┬──────────────┘
                           ▼
                   融合/重排（RRF 或 reranker）
                           ▼
                        Final TopK
```

这样能同时减少：

- 只靠关键词导致的“同义改写漏召回”
- 只靠向量导致的“术语/型号不精确”

## 7. Qdrant 到底干嘛

Qdrant 是向量数据库，负责：

1. 存向量 + payload
2. TopK 相似检索
3. 条件过滤（如 `session_id`、`source`、时间范围）
4. ANN 索引加速

它不负责：

- 训练模型
- 生成 embedding

```text
Qdrant ≈ 向量版 Elasticsearch
```

## 8. 你之前学的“标准 RAG”与当前项目的差异

你之前：

```text
文档 -> 语义 Embedding -> Chroma -> Retriever(similarity/MMR/threshold)
```

你当前：

```text
对话文本 -> HashEmbedder -> Qdrant -> TopK + threshold
```

所以差异不在“是否用向量库”，而在“向量是否语义化”。

bge-large-zh:
  dist(query, "AI选品商业化")  通常更近

## 10. 为什么会召回“打球”

### 原因 A：候选池机制

同一 `session_id` 下的历史内容都可能进候选池：

```text
M1: 下雨了/打球
M2: AI选品能赚钱吗
M3: 其他

query: AI选品能赚钱吗
-> HashEmbedder(query) 与 M1/M2/M3 比近邻
-> M1 可能被误判“够近”
```

### 原因 B：Hash 空间非语义

HashEmbedder 不能稳定保证“语义近的文本向量也近”，所以 TopK 可能混入无关内容。

### 结果链路

```text
Hash近邻偏差 -> TopK混入无关记忆 -> 注入Memories -> LLM被带偏
```

---

## 11. 针对你项目的落地建议

1. 先把 Embedder 切到 `bge-large-zh`（本地服务）
2. Qdrant 保持不变，只确保维度一致
3. 观察召回质量后，再考虑 MMR / Hybrid

再强调一次：

**同样的 Qdrant，只要 embedding 从 hash 换成语义模型，召回通常会明显改善。**
